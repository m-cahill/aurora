# M34 — CI, QA, and supply-chain hardening

**Timeframe:** Merged to `main` 2026-05-06 (PR #35, merge commit `d894ad95…`); closeout **PR** **#42** (merge `cab94a72…`).

## Summary

M34 enforced **strict Mypy** on **`src/aurora/`**, **`pip-audit`** over the **full** dev install in **`ci` / `repo-safety`**, and weekly **Dependabot** for **pip** and **GitHub Actions**. Pinned **`mypy`**, **`pip-audit`**, and **`setuptools`** landed in **`requirements-dev.txt`** with **`[tool.mypy]`** in **`pyproject.toml`**. Workflow **job identity** stayed the same; **no** `continue-on-error` on enforcement steps. **`DEVELOPMENT.md`**, **`AURORA_OPERATING_MANUAL.md`**, and **`README.md`** were aligned; **`src/aurora/arb/validator.py`** received **non-behavioral** typing adjustments for strict checking.

## What changed

- **`.github/workflows/ci.yml`** (additive steps inside existing posture), **`.github/dependabot.yml`**, dev pins, Mypy config, minimal typing fixes.

## What it proved

- Green CI could mean **typed**, **audited** installs in a **reproducible** way—not only tests and coverage.

## What it did not prove

- Release tag or lock (**M39**); weakened gates; new ARB or runtime semantics.

## Key deliverables

- **Mypy** + **pip-audit** + **Dependabot** as required checks; documented operator expectations.

## Evidence (public)

- PR #35 and M34 closeout PR #42; run IDs in **Milestone M34** in [`aurora.md`](../aurora.md).

## Why it mattered

Public repos inherit **supply-chain** and **type** risk; M34 made those risks **visible and gated** without claiming application-level MediaPipe proof.
