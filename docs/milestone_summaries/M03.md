# M03 — Branch protection enforcement and CI governance hardening

**Timeframe:** Initialized 2026-03-26; merged to `main` (PR #3, merge commit `ce53b7c…`).

## Summary

M03 **hardened** the CI story: **full SHA pins** for third-party Actions, stable **`ci` / `repo-safety`** identity checks in the verifier, **`DEVELOPMENT.md`** for contributors, and **`ci-artifacts`** upload naming. Branch protection on `main` was configured to require the **`ci / repo-safety`** context (documented via GitHub API evidence in the ledger).

## What changed

- Verifier rules for workflow policy (no floating `ubuntu-latest` on enforcement workflows in the intended design).
- Tests expanded for verifier behavior.
- Contributor-facing documentation for the required check name.

## What it proved

- PRs could be **gated** on the same check name developers see locally.
- Supply-chain hygiene for Actions could be **encoded** in-repo.

## What it did not prove

- No runtime seams beyond what existed after M02/M04/M05 sequence (substrate arrives in M05).

## Key deliverables

- Hardened `verify_repo_state.py`, `DEVELOPMENT.md`, workflow policy alignment.

## Evidence (public)

- PR #3 merge; branch protection settings summarized in ledger (strict required checks).

## Why it mattered

Governance without **merge discipline** drifts. M03 tied **green CI** to **mergeability** for `main`.
