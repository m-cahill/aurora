# M22 — Bounded D1 native AUDIO_FROM_BYTES on NativeAudioDispatcher

**Timeframe:** Initialized 2026-04-01; merged to `main` 2026-04-01 (PR #23, merge `1985900…`).

## Summary

M22 extended **`NativeAudioDispatcher`** to handle **`AUDIO_FROM_BYTES`** with the **same** lifecycle as **`AUDIO_FROM_FILE`**. Raw bytes are interpreted as **structural** float32 lanes for tests — **not** codec decode. **Local-native** proof with a real prebuilt Tasks library was **not attempted** (documented as external/tooling absence, not a code defect).

## What changed

- `native_audio_dispatcher.py`, tests, doc pointers in `kernel_ingress_strategy.md` and `runtime_seam_framing.md`.

## What it proved

- Bytes and file paths can share one **dispatcher** implementation with honest **non-decode** semantics in tests.

## What it did not prove

- Decode correctness, graph/TFLite correctness, MediaPipe parity, or CI execution against a real binary.

## Key deliverables

- **`AUDIO_FROM_BYTES`** on `NativeAudioDispatcher` with frozen **`dispatch(token, payload, lib)`** arity.

## Evidence (public)

- PR #23; CI structural tests green.

## Why it mattered

Bytes ingress is required for many pipelines; M22 did it **without** pretending codec correctness existed.
